# Planner追補: Trackball Control Practice 専用依頼

## 結論
Trackball Control Practiceは「フッターは編集しない」前提で、ゲームエリア側のレイアウト再設計と、プレイ中の進行可視化、ターゲット視認性の改善を優先してください。

## 対象ページ
- games/trackball-controll-practice.html
- css/trackball-controll-practice.css
- js/trackball-controll-practice.js

## 制約（最優先）
- フッター（構造・文言・デザイン）は変更しない。
- 重なり問題は、ゲームページ側（main/app/playfieldの構造と余白、height設計）だけで解消する。
- 既存の共通ヘッダー/フッター仕様は維持する。

## 課題別の解決方針

### 課題① フッターとゲーム実施エリアが被る
1. ゲーム画面の縦サイズを固定値中心（560px）から、ビューポート基準の可変設計へ変更する。
2. プレイエリアの高さは「ヘッダー・情報パネル・フッター占有分」を差し引いた範囲で clamp する。
3. body の中央寄せ前提レイアウトを解除し、上から自然に積む通常フローに切り替える。
4. main/appの下余白を明示し、固定フッターと視覚衝突しない安全領域を確保する。

UI Designerには、次の2案を比較提示してほしい。
- 案A: 1画面内完結型（プレイエリアをやや小さくし、常にフッター非干渉）
- 案B: 縦スクロール許容型（情報を優先し、プレイエリアは可変で下に伸ばす）

### 課題② プレイ中に時間・進捗・終了タイミングが分かりづらい
1. 進捗表示を「数値 + プログレスバー」の二重化にする。
2. タイマーは強調表示し、状態ラベル（準備中 / プレイ中 / 一時停止 / クリア）を併置する。
3. 終了条件を常時表示する（例: 10ターゲット到達で終了）。
4. 残り回数を明示する（例: 残り3回）。
5. クリア直前（残り1-2回）で視覚強調し、終了タイミングの予測をしやすくする。

UI指示:
- モバイルでは「タイマー」「進捗」「残り回数」を1行で潰さず2段構成にする。
- ステータス文は色依存だけにせず、ラベル文言で意味が伝わる設計にする。

### 課題③ ターゲットが大きく、ホバー時に当たり判定が分かりづらい
1. ターゲットの標準サイズを縮小し、難易度ごとに段階差をつける。
2. hover/focus時に「外周リング」と「中心点」を表示し、当たり判定中心を明確化する。
3. カーソル接触時に見た目が反応する（色変化・軽いスケール・影変化）ようにする。
4. クリック可能状態と無効状態（停止中）で見た目を明確に分離する。

UI指示:
- hoverだけでなくfocus-visibleでも同等の視認性を確保する。
- 視認性向上の演出は200-300ms以内の控えめな動きに限定する。

## UI Designerへの提出物（今回依頼）
1. 上記3課題を同時に解消するレイアウト改善案を2案。
2. 案ごとのメリット/デメリット（実装負荷、モバイル安定性、視認性）。
3. 採用案1つと、その理由。
4. 実装者向けの具体メモ（どのDOMブロックをどう再配置するか、どの情報を常時表示するか）。

## 受け入れ基準
- フッターを一切編集せず、ゲームエリアとの重なりが発生しない。
- 初見ユーザーが「今どれだけ進んだか」「いつ終わるか」を3秒以内に理解できる。
- ターゲットの当たり判定位置をhover/focusで視覚的に判断できる。
- モバイル幅でも情報の欠落・詰まり・誤タップ誘発がない。
